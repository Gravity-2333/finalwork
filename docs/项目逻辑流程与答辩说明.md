# AI 学习助手项目逻辑流程与答辩说明

本文档用于项目展示和答辩说明，重点解释系统如何满足大作业要求，以及老师提问时可以如何说明核心流程。

## 一、项目整体定位

本项目实现的是一个“AI 学习助手”系统。用户通过人脸识别登录后，可以上传课程资料，系统会把资料切割并保存成本地专属知识库，再基于知识库自动生成课程大纲、章节学习内容、章节测验和错题归档。

系统采用前后端分离结构：

- 前端：Vue 3 + Vite，负责登录、人脸采集、资料上传、学习路径、测验复盘、语音控制和设置界面。
- 后端：FastAPI + SQLite，负责资料入库、知识库检索、LangChain 文档处理、LangGraph 流程编排、模型调用、人脸特征比对和讯飞语音识别代理。
- AI 编排：LangChain 负责资料加载与文本切割，LangGraph 负责编排“检索资料 -> 组装提示词 -> 调用模型 -> 解析结果”的流程，LangSmith 作为可选追踪能力，默认关闭。

## 二、对应大作业要求的实现说明

### 1. 上传课程资料，搭建本地专属知识库

用户在“知识库”页面选择课程资料上传，系统支持 `txt`、`md`、`docx`、`pdf` 等文本型资料。前端限制单批最多 20 个文件、单文件最大 25MB，避免一次上传过多导致请求异常。

后端接收到文件后会执行以下流程：

1. 保存上传文件到本地 `uploads` 目录。
2. 使用 LangChain Loader 按文件类型读取文本。
3. 对 PDF 等资料优先提取可用文本，过滤明显乱码片段。
4. 使用文本切割器把资料拆成多个 chunk。
5. 将文档信息、切片内容、来源文件和元数据写入本地 SQLite。

因此系统不是把资料直接一次性塞给大模型，而是先构建本地知识库。后续生成大纲、章节内容、测验题和知识库检索，都会从本地资料切片中取上下文。

答辩时可以强调：

- “专属”体现在资料由当前用户上传并保存在本地数据库中。
- “知识库”体现在资料被结构化切片保存，并支持关键词检索。
- AI 生成不是脱离资料自由发挥，而是依赖本地知识库上下文。

### 2. 智能梳理课程大纲，生成章节内容并记录进度

用户上传资料后，点击“生成课程学习路径”或通过语音说“初始化课程”，系统会进入课程初始化流程。

初始化流程如下：

1. 检查本地知识库是否已有资料。
2. 调用后端 `/api/outline` 生成课程大纲。
3. 优先从 PDF 目录或文本章节标题中识别真实章节结构。
4. 如果资料没有明确目录，再根据资料切片数量和文本规模估算章节数。
5. 使用 LangGraph 执行“检索上下文 -> 组装提示词 -> 调用模型 -> 解析大纲”。
6. 将章节标题、学习目标和顺序写入数据库。
7. 按章节依次生成学习内容，并保存 Markdown 格式正文。
8. 每章内容生成后，后台继续生成对应测验题。

前端会展示课程进度，例如资料数量、章节数量、整体进度、错题数量。章节内容生成后会标记学习进度，用户也可以进入学习路径页面查看每章内容。

答辩时可以说明：

- 对有目录的 PDF，系统优先尊重原始资料目录，避免固定生成少量章节。
- 对没有目录的资料，系统会根据内容规模估算章节数，并用模型补充合理大纲。
- 章节正文以 Markdown 渲染，便于展示标题、列表、重点解释等学习内容。

### 3. 自动出题、在线测验和错题归档

测验模块围绕章节展开。每个章节生成学习内容后，系统会在后台自动生成该章测验题。

测验生成流程：

1. 根据当前章节标题、学习目标、章节内容和知识库相关片段构造提示词。
2. 通过 LangGraph 调用当前配置的 AI Provider。
3. 要求模型返回结构化题目，包含题干、选项、正确答案、解析和题型标签。
4. 后端对题目进行规范化校验，确保前端可以稳定渲染。
5. 保存到 SQLite 的 quiz 表中。

用户点击“开始测验”时：

- 如果题目已经生成，进入答题页面。
- 如果题目还在后台生成，按钮会置灰或给出等待提示，避免进入空测验。
- 如果没有资料或没有章节，系统会提示先上传资料并初始化课程。

提交测验后：

1. 后端对比用户答案和标准答案。
2. 计算正确数、正确率和错题数。
3. 返回每道题的结果、正确答案、用户答案和解析。
4. 默认把做错的题加入错题归档。
5. 用户也可以手动把正确题加入错题，或从错题中移除。

界面上每道题下方直接显示解析和错题操作按钮，方便用户边看原题边复盘。

答辩时可以强调：

- 测验题不是固定写死，而是根据当前章节知识点动态生成。
- 错题归档不只是统计错误数量，还保存题目、答案、解析和复盘建议。
- 重新测验时会清空本轮答题结果，重新进入答题状态，避免旧结果残留。

### 4. 人脸识别核验登录，保障账号安全

系统进入主工作台前必须先经过人脸登录页面。用户输入账号名称后，可以录入或识别该账号对应的人脸。

人脸识别流程：

1. 前端通过浏览器摄像头获取视频画面。
2. 使用 `face-api.js` 加载人脸检测、关键点和识别模型。
3. 检测画面中是否只有一张人脸。
4. 从稳定的多帧画面中提取人脸 descriptor。
5. 额外生成 16x16 灰度哈希模板作为辅助特征。
6. 将 128 维深度人脸特征和灰度模板拼接为综合特征。
7. 后端保存账号对应的人脸模板。
8. 登录时重新采集当前人脸特征，与数据库模板进行距离比对。
9. 距离小于阈值才允许登录。

系统同时处理了多个体验细节：

- 检测或录入过程中按钮会锁定，避免重复点击造成并发错误。
- 检测有超时机制，超时会提示重新调整光线、角度或摄像头。
- 如果检测到多张人脸，会拒绝登录。
- 已登录用户可以重新录入授权人脸。
- 退出登录后必须重新人脸核验才能进入系统。

答辩时需要说明限制：

- 当前方案是真正基于人脸特征的账号核验，不是任意点击通过。
- 但它仍属于课程项目级实现，不是工业级活体检测系统。
- 目前不能完全防止高质量照片攻击；如果要进一步增强，可以加入眨眼、摇头、张嘴等活体动作检测，或接入专业活体检测 SDK。

### 5. 全系统兼容语音交互模式

系统提供“语音助手”页面，支持通过语音执行常用操作。当前语音识别支持两种模式：

- 讯飞 API 服务：默认模式，前端录音后发送给后端，由后端代理调用讯飞语音听写 WebAPI。
- 浏览器 Web Speech：可选模式，使用浏览器内置语音识别能力。

讯飞模式流程：

1. 用户选择麦克风输入设备。
2. 前端请求麦克风权限并开始录音。
3. 使用 AudioContext 实时显示输入音量。
4. 停止录音后，将音频重采样为 16kHz PCM。
5. 前端把 PCM 音频上传到后端 `/api/asr/recognize`。
6. 后端读取环境变量中的讯飞凭证。
7. 后端生成 WebSocket 鉴权地址，调用讯飞语音听写接口。
8. 后端接收识别文本并返回前端。
9. 前端根据识别文本匹配固定命令。
10. 命中命令后执行页面跳转或业务操作。

支持的语音能力包括：

- 打开知识库
- 打开学习路径
- 查看错题
- 打开模型设置
- 打开系统设置
- 初始化课程
- 开始测验
- 测试麦克风
- 获取云端模型
- 切换模型来源

系统也提供兜底能力：

- 语音识别失败时显示原因。
- 麦克风测试可以判断是否有权限、是否有输入设备、音量是否过低。
- 用户可以通过按钮或文本命令手动执行同样操作。

答辩时可以强调：

- 语音不是单纯装饰功能，而是和系统操作绑定。
- 有些语音命令只做页面跳转，有些会触发实际业务，如初始化课程、开始测验、测试模型连接。
- 语音识别密钥只在后端使用，不暴露给浏览器。

### 6. LangChain、LangGraph、LangSmith 技术栈

本项目对三项技术的使用方式如下：

LangChain：

- 用于加载 `txt`、`md`、`docx`、`pdf` 等资料。
- 用于文档切割，把长资料拆成适合检索和大模型输入的 chunk。

LangGraph：

- 用于组织 AI 生成流程。
- 核心节点包括：检索资料、组装提示词、调用模型。
- 大纲生成、章节内容生成、测验生成都围绕这个流程扩展。

LangSmith：

- 作为可选 tracing 工具。
- 默认关闭，避免没有密钥时影响系统运行。
- 用户在模型设置中启用追踪，并配置 `LANGSMITH_API_KEY` 后，后端才会设置对应环境变量并开启追踪。

## 三、核心业务流程

### 流程 1：从上传资料到生成完整课程

```text
用户上传资料
  -> 后端保存文件
  -> LangChain 读取文本
  -> 文本切割成 chunk
  -> 写入 SQLite 本地知识库
  -> 用户点击生成课程学习路径
  -> 后端生成课程大纲
  -> 保存章节列表
  -> 逐章生成学习内容
  -> 后台生成每章测验
  -> 前端展示进度和下一步建议
```

### 流程 2：知识库检索

```text
用户输入关键词
  -> 前端请求 /api/knowledge/search
  -> 后端在 SQLite chunk 中检索
  -> 返回命中文件、片段内容和来源信息
  -> 前端展示检索结果
```

该功能用于体现“本地专属知识库”不是只为生成内容服务，也可以被用户主动查询。

### 流程 3：章节学习

```text
用户进入学习路径
  -> 选择章节
  -> 若章节内容已生成，直接展示 Markdown 内容
  -> 若未生成，点击生成学习内容
  -> 后端检索该章相关知识库上下文
  -> AI 生成章节讲义
  -> 保存章节内容和进度
```

### 流程 4：在线测验和错题复盘

```text
章节测验已生成
  -> 用户开始测验
  -> 前端展示题目和选项
  -> 用户提交答案
  -> 后端批改并计算正确率
  -> 错题自动进入错题归档
  -> 前端逐题显示解析
  -> 用户可手动加入或取消错题归档
```

### 流程 5：人脸登录

```text
输入账号
  -> 打开摄像头
  -> 首次录入人脸模板
  -> 后端保存账号与人脸特征
  -> 下次登录重新采集人脸
  -> 后端计算模型距离和灰度模板差异
  -> 通过阈值校验后进入系统
```

### 流程 6：语音控制

```text
用户说出命令
  -> 前端录音或浏览器识别
  -> 得到文本
  -> 匹配固定命令
  -> 判断当前状态是否允许执行
  -> 执行页面跳转或业务操作
  -> 显示执行结果或失败原因
```

## 四、模型 Provider 设计

系统没有把模型接口写死，而是做成可切换 Provider：

- Mock：无密钥演示模式，适合课堂展示和无网络环境。
- 本地 Ollama：连接本机 Ollama 服务。
- 云端 Ollama：可获取并测试可用模型列表，过滤不适合当前项目的模型。
- 兼容接口：通用 OpenAI-compatible 方式，可填写 DeepSeek 等服务的地址、模型名和 Key。

API Key 支持两种方式：

- 手动输入：直接在前端填写 Key。
- 环境变量：前端只填写变量名，后端从系统环境变量读取真实 Key。

这样设计的原因：

- 课堂演示可以用 Mock。
- 本机部署可以用本地 Ollama。
- 有云端模型时可以切换云服务。
- DeepSeek 等服务可通过兼容接口接入，不需要单独写死成某一个厂商。

## 五、数据存储说明

项目使用 SQLite 保存核心数据：

- 上传文档记录
- 文档切片内容
- 课程章节
- 章节学习内容
- 测验题
- 答题记录
- 错题归档
- 人脸账号模板

这种设计适合课程项目：

- 部署简单，不需要额外数据库服务。
- 数据都在本地，能体现本地专属知识库。
- 方便演示、迁移和检查。

## 六、可向老师解释的关键问题

### 问：你的知识库是真正本地的吗？

是。上传文件会保存在本地目录，资料内容会被 LangChain 读取并切割，然后写入本地 SQLite。后续检索、大纲、章节内容、测验都基于这些本地切片。

### 问：为什么不直接把整个 PDF 发给大模型？

长 PDF 可能超过模型上下文限制，直接发送也不利于定位内容。系统先提取目录和文本，再切片入库。生成时按任务检索相关片段，这样更稳定，也更符合知识库系统思路。

### 问：大纲章节数怎么确定？

如果 PDF 有目录，优先读取 PDF 目录，按真实章节生成。如果没有目录，则根据文本中的章节标题和资料规模估算章节数，再由模型补充大纲内容，避免固定只生成少量章节。

### 问：人脸识别是不是假的？

不是。前端使用 `face-api.js` 提取人脸深度特征，并叠加灰度模板辅助特征。后端保存账号模板，登录时计算当前人脸与模板的距离，未达到阈值不会通过。

### 问：能不能防照片？

当前课程项目实现可以进行人脸特征比对，但不是完整工业级活体检测。要防照片攻击，需要加入眨眼、摇头、张嘴等动作活体检测，或接入专业活体检测 SDK。

### 问：语音识别为什么要走后端？

讯飞 API 需要 `APISecret` 做签名，不能放在浏览器中，否则会泄露密钥。因此前端只负责录音，后端负责鉴权和调用讯飞接口。

### 问：LangGraph 用在哪里？

LangGraph 用在 AI 生成流程编排中，把生成任务拆成检索、组装 Prompt、调用模型等节点。大纲生成、章节生成和测验生成都复用这个流程。

### 问：没有 API Key 时能不能演示？

可以。系统提供 Mock Provider，可以离线演示完整流程。真实模型可通过本地 Ollama、云端 Ollama 或兼容接口切换。

### 问：错题归档如何工作？

提交测验后，后端自动把错题加入错题归档。前端也允许用户手动把正确题加入错题，或取消归档。这样既支持自动复盘，也支持用户自主整理重点题。

## 七、项目亮点总结

1. 不是简单聊天机器人，而是围绕课程资料构建学习流程。
2. 本地知识库可上传、切割、检索、删除和清空。
3. 大纲、章节内容、测验、错题形成完整学习闭环。
4. 人脸识别登录满足账号安全核验要求。
5. 语音控制可直接触发系统操作，提高交互完整度。
6. Provider 架构可适配 Mock、本地模型、云端模型和 DeepSeek 等兼容接口。
7. LangSmith 可选开启，保证无密钥环境下系统仍能正常运行。
