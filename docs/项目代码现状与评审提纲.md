# AI 学习助手项目代码现状与评审提纲

本文档用于发给网页 GPT 或其他代码审查助手，请其基于当前项目现状给出代码结构、功能完整性、前端设计风格和用户体验优化建议。

## 1. 项目定位

项目名称：AI 学习助手

项目类型：课程大作业，可本地运行、截图、生成报告。

核心目标：

- 上传课程资料并构建本地知识库。
- 基于知识库生成课程大纲、章节学习内容、在线测验。
- 记录学习进度、错题归档和复盘。
- 支持人脸识别登录，满足“账号登录采用人脸识别核验方式”的作业要求。
- 支持语音指令控制常用功能。
- 支持多种 AI Provider，并保留 Mock 模式以便无密钥演示。

## 3. 后端功能现状

### 3.1 API 入口

`backend/app/main.py` 定义主要接口：

- `GET /api/health`
- `GET /api/documents`
- `POST /api/documents/upload`
- `POST /api/documents/upload-batch`
- `DELETE /api/documents/{document_id}`
- `DELETE /api/documents`
- `POST /api/outline`
- `GET /api/chapters`
- `POST /api/chapters/{chapter_id}/content`
- `POST /api/chapters/{chapter_id}/quiz`
- `GET /api/chapters/{chapter_id}/quiz`
- `POST /api/chapters/{chapter_id}/submit`
- `GET /api/wrong-answers`
- `POST /api/provider/test`
- `POST /api/provider/cloud-ollama/models`
- `GET /api/face/profile`
- `POST /api/face/enroll`
- `POST /api/face-login`

上传限制：

- 单文件最大 25MB。
- 单批最多 20 个文件。
- 支持 `.txt`、`.md`、`.docx`、`.pdf`。
- 暂不支持图片和 PPT，因为项目没有 OCR 流程。

### 3.2 知识库处理

文件：`backend/app/knowledge.py`

当前逻辑：

- 上传文件保存到 `uploads/`。
- 文件名做基础清理，避免特殊字符。
- 同名同内容文件直接复用。
- 同名不同内容文件自动追加编号。
- txt/md 使用 `TextLoader`。
- pdf 使用 `PyPDFLoader` 提取文本。
- docx 使用 `python-docx` 读取段落。
- 使用 `RecursiveCharacterTextSplitter(chunk_size=650, chunk_overlap=90)` 切片。
- chunk 保存到 SQLite。
- 简单检索使用关键词计数加文本长度权重，不是向量检索。

删除资料时会同步清空章节、测验、错题状态。

### 3.3 LangGraph 工作流

文件：`backend/app/workflow.py`

当前图结构：

```text
retrieve -> compose -> generate -> END
```

用途：

- `outline`：读取知识库前若干 chunk，生成 4 个章节。
- `chapter`：按章节标题和目标检索相关 chunk，生成章节内容。
- `quiz`：当前没有走完整 graph，而是直接组合 prompt 调 Provider。

保护逻辑：

- 未上传资料时调用大纲、章节、测验会报错：请先上传课程资料。
- AI 返回的大纲会被解析成 4 个章节；如果解析失败，会用备用大纲。
- AI 返回的测验期望 JSON 数组；解析失败会用备用题。
- 章节内容会清理模型寒暄、代码围栏和多余标题。

默认 prompt 支持用户在前端设置里自定义：

- `{{context}}`
- `{{chapter_title}}`
- `{{chapter_objective}}`

### 3.4 Provider 架构

文件：`backend/app/providers.py`

当前支持：

- `mock`：离线演示，不需要 API Key。
- `local_ollama`：默认 `http://localhost:11434`，调用 `/api/chat`。
- `cloud_ollama`：按 OpenAI-compatible `/chat/completions` 调用，默认 base_url 为 `https://ollama.com/v1`。
- `openai_compatible`：通用 OpenAI 兼容接口，可用于 DeepSeek 等服务。

API Key 支持两种模式：

- 前端手动输入 `api_key`。
- 输入环境变量名 `api_key_env`，后端从系统环境变量读取。

调用失败时：

- 正式生成功能默认允许 fallback 到 Mock，并返回 warning。
- Provider 测试接口不 fallback，失败会直接返回错误，便于用户定位配置问题。

云端 Ollama 模型列表：

- 调 `/models` 获取模型。
- 过滤和排序更适合当前文本任务的模型。
- 对候选模型逐个发送小请求测试，只返回可用模型。

### 3.5 人脸识别

文件：

- `frontend/src/components/FaceGate.vue`
- `backend/app/face_auth.py`

实现方式：

- 前端用 face-api.js 加载模型：
  - tiny face detector
  - face landmark 68
  - face recognition
- 摄像头采集画面。
- 每次录入或登录前连续采集 5 帧。
- 对 128 维 face-api descriptor 做平均，降低单帧波动。
- 额外生成 16x16 灰度二值模板作为弱辅助。
- 后端保存组合 descriptor：128 维模型特征 + 256 维灰度模板。

当前判定策略：

- 主要看 face-api 欧氏距离，默认阈值 `0.38`。
- 如果模型距离足够好，例如低于 `0.30`，则以深度人脸特征为主，灰度模板只作为参考。
- 如果模型距离接近阈值，则灰度模板会更严格参与判断。
- 这样可以减少换环境、换光线、换摄像头时被灰度模板误拒。

账号逻辑：

- 支持不同账号分别录入人脸。
- 未登录状态下，已录入账号不能随意覆盖模板。
- 登录后可重录当前账号人脸，后端使用 replace token 控制权限。
- 支持旧模板升级提示。

限制：

- 这是本地课堂项目级别的人脸识别，不是工业级活体检测。
- 不能可靠防止照片攻击。
- 未使用眨眼、转头、深度相机、红外、挑战动作等强活体检测。

### 3.6 SQLite 数据结构

文件：`backend/app/database.py`

表：

- `documents`
- `chunks`
- `chapters`
- `quizzes`
- `wrong_answers`
- `face_profiles`

目前没有用户表、角色表和登录 session 表。人脸登录状态主要在前端内存中维护；后端只负责人脸模板与比对。

### 4.2 知识库页面

### 4.3 学习路径页面

功能：

文件：`QuizPanel.vue`

功能：

- 展示当前章节测验。

### 4.7 语音助手

文件：

- `frontend/src/useVoice.js`
- `App.vue` 中 voice page 模板

功能：

- Web Speech API 识别中文命令。
- 支持开始/再次点击停止。
- 停止时会尝试处理最后识别到的文本。
- 识别到命令关键词会自动停止并执行。
- 支持麦克风测试：
  - 检查权限状态。
  - 检查输入设备数量。
  - 读取音量 RMS，提示音量偏低/正常/清晰。

命令分类：

- 页面跳转：
  - 上传资料
  - 打开知识库
  - 打开学习路径
  - 打开测验复盘
  - 查看错题
  - 模型设置
  - 系统设置
- 自动执行：
  - 生成课程大纲
  - 生成章节内容
  - 开始测验
  - 测试模型连接
  - 获取云端模型
  - 麦克风测试
  - 切换 Mock 模型
  - 切换本地 Ollama
  - 切换云端 Ollama
  - 切换兼容模型

已知限制：

- Web Speech API 在 Edge/Chrome 上依赖浏览器语音服务，网络不可用时会报 `network`。
- 不是完全本地语音识别。
- 命令匹配是关键词包含匹配，没有语义理解。

## 8. 主要设计取舍

### 8.1 为什么保留 Mock Provider

老师或演示机器可能没有 DeepSeek/Ollama API Key，也可能没有本地 Ollama 服务。Mock 能保证完整流程可演示。

### 8.2 为什么不支持图片和 PPT 上传

当前没有 OCR 模块。为了避免用户上传图片或 PPT 后 AI Provider 不支持视觉识别，项目限制为文本类资料和 PDF/DOCX 文本提取。

### 8.3 为什么不用向量数据库

当前作业优先保证可运行、可解释、交付稳定。检索使用 SQLite 全量 chunk + 关键词计数，简单但效果有限。后续可以替换为 embedding + FAISS/Chroma。

### 8.4 为什么人脸识别不是工业级安全

浏览器端 face-api.js 能满足课堂项目演示和账号绑定流程，但缺少真正活体检测、防照片攻击、防重放攻击和服务端图像证据链。
